# 外呼通话文本质检 Skill 设计文档

**日期**: 2026-06-01
**状态**: Draft

---

## 1. 项目概述

为 Claude Code 构建一个质检 skill，用于外呼通话对话文本的自动化质量检测。用户选择质检规则文件和源数据文件后，skill 以科学的质量体系高效完成质检，并输出质检报告。

**核心目标**：
- 支持 1w-5w 条对话数据的大规模质检
- 基于外部质检规则文件（由问题挖掘 skill 生成）
- 每条 case 打标结果：问题类型、危害程度、改进建议
- 非"有意向"对话的意向归因分析

## 2. 系统架构

**Prompt 驱动 + 少量 Python I/O 混合架构**

### 2.1 核心理念：统一框架，两套规则

**质检和归因本质上是同一件事：给对话打标，只是打的标签不同。**

- **通道一（合规检测）**：注入合规规则 → 输出"违规/不违规"
- **通道二（归因分析）**：注入归因规则 → 输出"什么因素导致没谈成"

框架（身体）完全一致：拆分 → 分发 → 收集 → 校验 → 汇总
区别仅在于注入的规则包（灵魂）。

### 2.2 统一框架流程

```
┌─────────────────────────────────────────────┐
│              Supervisor Agent               │
│  1. 加载规则包（合规 or 归因）               │
│  2. 调用 Python 读取数据 → 拆分批次         │
│  3. 并发 5 个 Worker Agent → 逐批打标        │
│  4. 收集结果 → 完整性校验                    │
│  5. 如有遗漏 → 重新分发                      │
│  6. 调用 Python 输出结果                     │
└─────────────────────────────────────────────┘
         ↓ 两次运行，不同的规则包
┌────────────────────────┐  ┌────────────────────────┐
│ 通道一: 合规检测        │  │ 通道二: 归因分析        │
│ 规则包: rules.md       │  │ 规则包: attribution.md  │
│ 数据: 全量对话          │  │ 数据: 非"有意向"对话    │
│ 输出: 合规检测结果      │  │ 输出: 归因分析报告      │
└────────────────────────┘  └────────────────────────┘
```

### 2.3 职责分工

**Python 负责**：
- 读取 Excel 数据源（id、时间、对话文本 JSON、意向结果）
- 按批次拆分数据（每批 100 条）
- 写入质检报告 Excel（合规检测 sheet + 归因分析 sheet）
- 进度文件读写与校验

**LLM (Claude) 负责**：
- 解析规则文件（合规规则 or 归因规则）
- 逐批打标：理解对话 + 按规则判断 + 输出结果
- 交叉验证：随机抽样 1-2% 做 double-check

### 2.4 架构分层

```
┌─────────────────────────────────────────┐
│           Supervisor Agent              │
│  (规则加载、数据拆分、结果收集、校验)     │
├────────────┬────────────┬───────────────┤
│ Worker 1   │ Worker 2   │ ... Worker N  │
│ Batch打标  │ Batch打标  │   Batch打标   │
├────────────┴────────────┴───────────────┤
│          Python I/O 层                  │
│  (Excel读取/拆分、批次分发、报告输出)     │
├─────────────────────────────────────────┤
│  规则层: rules.md / attribution.md      │
│  数据层: .xlsx  (外部输入)              │
└─────────────────────────────────────────┘
```

### 2.5 执行流程

```
通道一：合规检测
1. Supervisor: 解析 rules.md → 提取规则元数据
2. Supervisor: 调用 Python 读取 Excel → 获取全量对话清单
3. Supervisor: 拆分批次 → 每批 100 条
4. Supervisor: 并发 5 个 Worker Agent → 每批次合规检测
5. Worker: 逐条对话 × 逐条合规规则 → 输出打标结果
6. Supervisor: 收集结果 → 完整性校验（数量+格式+交叉验证）
7. Supervisor: 如有遗漏/异常 → 重新分发
8. Supervisor: 调用 Python 写入合规检测结果

通道二：归因分析（同一框架，注入归因规则）
9.  Supervisor: 过滤非"有意向"对话 → 拆分批次
10. Supervisor: 注入归因规则 → 并发 Worker Agent
11. Worker: 逐条对话 × 逐条归因规则 → 输出归因结果
12. Supervisor: 收集结果 → 完整性校验
13. Supervisor: 调用 Python 追加归因分析 sheet
```

## 3. 规则解析

### 3.1 规则文件格式

解析 rules.md 格式的规则文件，结构为：

```markdown
## R01: 规则名称

**严重程度**: 高/中/低
**发现次数**: 数字

**描述**: 规则描述文本

**检测逻辑**: 检测条件描述

**典型案例**:
- 案例1
- 案例2
```

### 3.2 解析结果

每条规则提取为结构化字段：
- `rule_id`: R01, R02, ...
- `name`: 规则名称
- `severity`: 高/中/低
- `description`: 规则描述
- `detection_logic`: 检测逻辑
- `examples`: 典型案例列表

### 3.3 规则包

所有规则打包为 JSON 结构体，随 Prompt 一起分发给 Worker Agent：

```json
{
  "rules": [
    {
      "rule_id": "R01",
      "name": "无视用户明确拒绝",
      "severity": "高",
      "description": "...",
      "detection_logic": "..."
    }
  ]
}
```

## 4. 批次与并发

### 4.1 批次参数

- **每批 100 条对话**: 约 68,000 字符（0.07M），上下文中非常安全
- **每次并发 5 个 Worker Agent**: Claude Code 能稳定处理的并发数
- **1w 条数据** = 100 批，约 20 轮并发

### 4.2 Worker 防偷懒 Harness 约束

**不靠 Prompt 喊话，用工程结构强制认真**：

1. **逐条输出强制**：Worker 必须对每条对话输出独立结果（包含 `id`），不能批量概括。Supervisor 收到结果后校验：结果数量 == 批次数量？每条都有唯一 `id`？缺少任一校验不通过。

2. **规则逐条遍历**：Worker 内部按"规则清单"逐条检查，每条规则过完后在中间变量中打勾（记录已检查的规则 ID）。最终输出时附带 `rules_checked: ["R01", "R02", ...]` 字段，证明所有规则都被遍历过。

3. **抽检内嵌**：Worker 输出时随机附带 3-5 条对话的详细分析过程（不仅给结论，还写出"我为什么判断这条违规/不违规"的推理链）。Supervisor 可以快速扫一眼判断 Worker 是否认真执行。

### 4.3 Worker Prompt 模板

每个 Worker 收到：
1. 规则包（JSON）
2. 批次数据（100 条对话的 JSON 格式）
3. 输出格式要求

输出格式（JSON）：
```json
{
  "batch_id": 1,
  "rules_checked": ["R01", "R02", "R03", ...],
  "spot_check_details": [
    {
      "id": "11126472134",
      "reasoning": "用户说'暂时不考虑'后，AI 继续推送岗位信息，触发 R01..."
    }
  ],
  "results": [
    {
      "id": "11126472134",
      "status": "pass",
      "violations": []
    },
    {
      "id": "11252027824",
      "status": "violation",
      "violations": [
        {
          "rule_id": "R01",
          "severity": "高",
          "evidence": "用户: 嗯，暂时不考虑了啊，谢谢 | AI继续: ...",
          "suggestion": "用户明确拒绝后应立即礼貌结束对话"
        }
      ]
    }
  ]
}
```

**输出量控制**：
- 无违规的对话只记录 `status: "pass"` + 空 `violations`，不输出冗余文本
- 有违规的对话才展开详细证据和片段
- 避免 100 条 × 13 条规则的冗余输出撑爆上下文

## 5. 校验机制

### 5.1 数量校验
- Supervisor 维护原始 ID 清单
- Worker 返回结果时带处理过的 ID 列表
- 比对：收到 ID 数 == 原始 ID 数？无重复？无遗漏？
- 不匹配时重新分发遗漏批次

### 5.2 格式校验
- 每条结果必须符合预设 JSON 结构
- 格式异常的批次要求重新质检

### 5.3 交叉验证（Harness 范式）

**分层抽样**：
- 按违规/无违规比例抽样。如整体违规率 15%，则从违规组抽 2%、无违规组抽 1%
- 确保两类对话都有覆盖，不是纯随机

**规则级对齐对比**：
- 不是看"两次结果是否完全一致"，而是看"同一条对话同一个规则，两次判断是否一致"
- 不一致的标记出来，由 Supervisor 做最终裁决
- 差异率 = 不一致的规则判断数 / 总对比规则数

**动态调整**：
- 差异率 < 5%：正常，继续
- 差异率 5%-10%：可疑，标记需人工复核
- 差异率 > 10%：Worker 整体质量有问题，扩大交叉验证比例到 5%

### 5.4 进度追踪
Supervisor 维护 `progress.json`：
```json
{
  "total_ids": 10000,
  "completed_ids": [...],
  "failed_batches": [],
  "batch_status": {"1": "done", "2": "done", ...},
  "cross_check_results": {}
}
```

## 6. 归因分析（同一框架，不同规则）

### 6.1 触发条件

通道一（合规检测）全部完成后自动触发。

### 6.2 分析对象

所有意向结果 ≠ "A(有意向)" 的对话。

### 6.3 归因规则

归因规则与合规规则格式相同（Markdown 结构化），但内容不同：
- 合规规则：判断是否违规（如"无视用户拒绝"）
- 归因规则：判断对话中什么因素导致没谈成（如"未介绍岗位亮点"、"语气过于机械"）

归因规则可以独立生成和更新，不需要改框架。

### 6.4 输入

- 非"有意向"对话的对话文本
- 通道一的质检结果（这些对话被标记了哪些问题）
- 意向结果标签（B/C/E/F/I 等）

### 6.5 分析内容

1. **问题关联**：哪些问题类型和"无意向"强相关？
2. **轮次分析**：对话在哪个轮次开始偏离？
3. **高频模式**：哪些因素是"无意向"的高频原因？
4. **对比分析**：对比"有意向"组 vs"其他"组的对话模式差异

### 6.6 输出

独立 sheet 的归因报告，包含：
- 归因统计（各因素占比）
- 典型案例（每类归因 3-5 个典型案例）
- 改进建议

|----|------|----------|----------|----------|----------|----------|----------|

- 违规率

- 严重程度分布

## 8. Skill 结构

```
~/.agents/skills/auto-qc/
├── SKILL.md                    # Skill 主文件（prompt + 流程定义）
├── templates/
│   ├── worker-prompt.md        # Worker Agent 质检模板
│   └── attribution-prompt.md   # 归因分析模板
└── scripts/
    ├── data_loader.py          # Excel 读取 + 批次拆分 + 进度管理
    └── report_writer.py        # 报告 Excel 输出 + 临时文件清理
```

### 8.1 用户交互

```
/auto-qc --data <excel路径> --rules <rules.md路径> [--attribution]
```

- `--data`: 源数据 Excel 路径（必需）
- `--rules`: 规则文件路径（必需）
- `--attribution`: 启用归因分析（可选，默认启用）

## 9. 错误处理

- Excel 读取失败 → 提示用户检查文件格式
- 规则解析失败 → 提示用户检查规则格式
- Worker Agent 超时/失败 → 自动重试（最多 3 次）
- 交叉验证差异率 > 5% → 标记为可疑，需要人工复核
- 进度文件损坏 → 从头开始或从最后一个完整批次恢复

### 10.2 关键节点汇报

Supervisor 在以下节点输出简短汇报：
- "已加载 X 条对话，拆分为 Y 批"
- "已完成 10/100 批（10%）"
- "交叉验证中（抽样 N 条）"
- "归因分析中"
- "报告生成完成：xxx"

用户可以随时中断查看进度文件。

## 11. 失败重试策略

- Worker Agent 失败/超时/输出格式异常 → 自动重试，最多 3 次
- 3 次都失败的批次 → 记录到 `failed_batches.json`
- 不阻塞整体流程，最后汇总报告时提醒用户
- `failed_batches.json` 示例：
  ```json
  {
    "failed_batches": [
      {"batch_id": 3, "error": "timeout", "retries": 3},
      {"batch_id": 47, "error": "invalid_output", "retries": 3}
    ]
  }
  ```

## 15. 核心约束

**LLM 主导原则**：
- Worker 质检环节（通道一合规检测 + 通道二归因分析）必须由大模型主导完成
- 禁止用 Python 代码替代 LLM 做质检判断
- Python 仅负责 I/O 操作：Excel 读取、批次拆分、进度管理、报告输出
- 质检的"理解对话、判断违规、输出打标、归因分析"全部由 Claude 的 sub-agent 机制完成
- SKILL.md 的设计应引导 Claude 合理使用 sub-agent 分发任务，而非写代码自动处理

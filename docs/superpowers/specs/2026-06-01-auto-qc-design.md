# 外呼通话文本质检 Skill 设计文档

**日期**: 2026-06-01
**状态**: Draft

---

## 1. 项目概述

为 Claude Code 构建一个质检 skill，用于外呼通话对话文本的自动化质量检测。用户选择质检规则文件和源数据文件后，skill 以科学的质量体系高效完成质检，并输出质检报告。

**核心目标**：
- 支持 1w-5w 条对话数据的大规模质检
- 基于外部质检规则文件（由问题挖掘 skill 生成）
- 每条 case 打标结果：问题类型、危害程度、改进建议
- 非"有意向"对话的意向归因分析

---

## 2. 系统架构

**Prompt 驱动 + 少量 Python I/O 混合架构**

### 2.1 核心理念：统一框架，两套规则

**质检和归因本质上是同一件事：给对话打标，只是打的标签不同。**

- **通道一（合规检测）**：注入合规规则 → 输出"违规/不违规"
- **通道二（归因分析）**：注入归因规则 → 输出"什么因素导致没谈成"

框架（身体）完全一致：拆分 → 分发 → 收集 → 校验 → 汇总
区别仅在于注入的规则包（灵魂）。

### 2.2 统一框架流程

```
┌─────────────────────────────────────────────┐
│              Supervisor Agent               │
│  1. 加载规则包（合规 or 归因）               │
│  2. 调用 Python 读取数据 → 拆分批次         │
│  3. 并发 5 个 Worker Agent → 逐批打标        │
│  4. 收集结果 → 完整性校验                    │
│  5. 如有遗漏 → 重新分发                      │
│  6. 调用 Python 输出结果                     │
└─────────────────────────────────────────────┘
         ↓ 两次运行，不同的规则包
┌────────────────────────┐  ┌────────────────────────┐
│ 通道一: 合规检测        │  │ 通道二: 归因分析        │
│ 规则包: rules.md       │  │ 规则包: attribution.md  │
│ 数据: 全量对话          │  │ 数据: 非"有意向"对话    │
│ 输出: 合规检测结果      │  │ 输出: 归因分析报告      │
└────────────────────────┘  └────────────────────────┘
```

### 2.3 职责分工

**Python 负责**：
- 读取 Excel 数据源（id、时间、对话文本 JSON、意向结果）
- 按批次拆分数据（每批 100 条）
- 写入质检报告 Excel（合规检测 sheet + 归因分析 sheet）
- 进度文件读写与校验

**LLM (Claude) 负责**：
- 解析规则文件（合规规则 or 归因规则）
- 逐批打标：理解对话 + 按规则判断 + 输出结果
- 交叉验证：随机抽样 1-2% 做 double-check

### 2.4 架构分层

```
┌─────────────────────────────────────────┐
│           Supervisor Agent              │
│  (规则加载、数据拆分、结果收集、校验)     │
├────────────┬────────────┬───────────────┤
│ Worker 1   │ Worker 2   │ ... Worker N  │
│ Batch打标  │ Batch打标  │   Batch打标   │
├────────────┴────────────┴───────────────┤
│          Python I/O 层                  │
│  (Excel读取/拆分、批次分发、报告输出)     │
├─────────────────────────────────────────┤
│  规则层: rules.md / attribution.md      │
│  数据层: .xlsx  (外部输入)              │
└─────────────────────────────────────────┘
```

### 2.5 执行流程

```
通道一：合规检测（独立运行）
1. Supervisor: 解析 rules.md → 提取规则元数据
2. Supervisor: 调用 Python 读取 Excel → 获取全量对话清单 → 预处理对话格式
3. Supervisor: 拆分批次 → 每批 100 条
4. Supervisor: 并发 5 个 Worker Agent → 每批次合规检测
5. Worker: 逐条对话 × 逐条合规规则 → 输出打标结果
6. Supervisor: 收集结果 → 完整性校验（数量+格式+交叉验证）
7. Supervisor: 如有遗漏/异常 → 重新分发
8. Supervisor: 调用 Python 写入合规检测结果

通道二：归因分析（独立运行，同一框架，注入归因规则）
9.  Supervisor: 过滤非"有意向"对话 → 拆分批次
10. Supervisor: 注入归因规则 → 并发 Worker Agent
11. Worker: 逐条对话 × 逐条归因规则 → 输出归因结果
12. Supervisor: 收集结果 → 完整性校验
13. Supervisor: 调用 Python 追加归因分析 sheet
```

### 2.6 运行模式

用户通过命令行参数控制运行模式：

| 命令 | 行为 |
|------|------|
| `/auto-qc --data <路径> --rules <路径>` | 默认全量运行：合规检测 + 归因分析 |
| `/auto-qc --data <路径> --rules <路径> --no-attribution` | 仅合规检测 |
| `/auto-qc --data <路径> --attribution-only` | 仅归因分析（使用内置归因规则） |

**归因分析是内置能力**：默认开启，无需指定规则文件，使用 `attribution-rules.md` 内置归因规则。用户可通过 `--no-attribution` 关闭。

合规检测规则必须由用户提供（`--rules`），框架不做假设。

## 3. 规则解析

### 3.1 规则文件格式

解析 rules.md 格式的规则文件，结构为：

```markdown
## R01: 规则名称

**严重程度**: 高/中/低
**发现次数**: 数字

**描述**: 规则描述文本

**检测逻辑**: 检测条件描述

**典型案例**:
- 案例1
- 案例2
```

### 3.2 解析结果

每条规则提取为结构化字段：
- `rule_id`: R01, R02, ...
- `name`: 规则名称
- `severity`: 高/中/低
- `description`: 规则描述
- `detection_logic`: 检测逻辑
- `examples`: 典型案例列表

### 3.3 规则包

所有规则打包为 JSON 结构体，随 Prompt 一起分发给 Worker Agent：

```json
{
  "rules": [
    {
      "rule_id": "R01",
      "name": "无视用户明确拒绝",
      "severity": "高",
      "description": "...",
      "detection_logic": "..."
    }
  ]
}
```

## 4. 批次与并发

### 4.1 批次参数

- **每批 100 条对话**: 约 68,000 字符（0.07M），上下文中非常安全
- **每次并发 5 个 Worker Agent**: Claude Code 能稳定处理的并发数
- **1w 条数据** = 100 批，约 20 轮并发

### 4.2 Worker 防偷懒 Harness 约束

**不靠 Prompt 喊话，用工程结构强制认真**：

1. **逐条输出强制**：Worker 必须对每条对话输出独立结果（包含 `id`），不能批量概括。Supervisor 收到结果后校验：结果数量 == 批次数量？每条都有唯一 `id`？缺少任一校验不通过。

2. **规则逐条遍历**：Worker 内部按"规则清单"逐条检查，每条规则过完后在中间变量中打勾（记录已检查的规则 ID）。最终输出时附带 `rules_checked: ["R01", "R02", ...]` 字段，证明所有规则都被遍历过。

3. **抽检内嵌**：Worker 输出时随机附带 3-5 条对话的详细分析过程（不仅给结论，还写出"我为什么判断这条违规/不违规"的推理链）。Supervisor 可以快速扫一眼判断 Worker 是否认真执行。

### 4.3 Worker Prompt 模板

每个 Worker 收到：
1. 规则包（JSON）
2. 批次数据（100 条对话的 JSON 格式）
3. 输出格式要求

输出格式（JSON）：
```json
{
  "batch_id": 1,
  "rules_checked": ["R01", "R02", "R03", ...],
  "spot_check_details": [
    {
      "id": "11126472134",
      "reasoning": "用户说'暂时不考虑'后，AI 继续推送岗位信息，触发 R01..."
    }
  ],
  "results": [
    {
      "id": "11126472134",
      "status": "pass",
      "violations": []
    },
    {
      "id": "11252027824",
      "status": "violation",
      "violations": [
        {
          "rule_id": "R01",
          "severity": "高",
          "evidence": "用户: 嗯，暂时不考虑了啊，谢谢 | AI继续: ...",
          "suggestion": "用户明确拒绝后应立即礼貌结束对话"
        }
      ]
    }
  ]
}
```

**输出量控制**：
- 无违规的对话只记录 `status: "pass"` + 空 `violations`，不输出冗余文本
- 有违规的对话才展开详细证据和片段
- 避免 100 条 × 13 条规则的冗余输出撑爆上下文

## 5. 校验机制

### 5.1 数量校验
- Supervisor 维护原始 ID 清单
- Worker 返回结果时带处理过的 ID 列表
- 比对：收到 ID 数 == 原始 ID 数？无重复？无遗漏？
- 不匹配时重新分发遗漏批次

### 5.2 格式校验
- 每条结果必须符合预设 JSON 结构
- 格式异常的批次要求重新质检

### 5.3 交叉验证（Harness 范式）

**分层抽样**：
- 按违规/无违规比例抽样。如整体违规率 15%，则从违规组抽 2%、无违规组抽 1%
- 确保两类对话都有覆盖，不是纯随机

**规则级对齐对比**：
- 不是看"两次结果是否完全一致"，而是看"同一条对话同一个规则，两次判断是否一致"
- 不一致的标记出来，由 Supervisor 做最终裁决
- 差异率 = 不一致的规则判断数 / 总对比规则数

**动态调整**：
- 差异率 < 5%：正常，继续
- 差异率 5%-10%：可疑，标记需人工复核
- 差异率 > 10%：Worker 整体质量有问题，扩大交叉验证比例到 5%

### 5.4 进度追踪
Supervisor 维护 `progress.json`：
```json
{
  "total_ids": 10000,
  "completed_ids": [...],
  "failed_batches": [],
  "batch_status": {"1": "done", "2": "done", ...},
  "cross_check_results": {}
}
```

## 6. 归因分析（同一框架，不同规则）

### 6.1 触发条件

通道一（合规检测）全部完成后自动触发。

### 6.2 分析对象

所有意向结果 ≠ "A(有意向)" 的对话。

### 6.3 归因规则

系统内置一套默认归因规则模板，同时支持外部规则文件覆盖。

**内置归因规则示例**：
- AI 未介绍岗位关键信息（薪资、地点、公司）
- 对话未识别用户兴趣点，未做针对性引导
- 回复过于模板化/机械，缺乏人情味
- 未正面回答用户追问，导致信任流失
- 节奏把控不当（推进过快/过慢）
- 结束话术过于敷衍，未约定下次联系

归因分析采用 5 Why 形式（逐步追问到根因，但不强制到第 5 层）：
- "为什么没谈成？" → "用户提出追问时 AI 未正面回答"
- "为什么 AI 未正面回答？" → "对话状态机跳转到错误的话术模板"
- "为什么跳转到错误话术？" → "用户回答不在预期分支内，系统 fallback 到兜底话术"

归因规则与合规规则格式相同（Markdown 结构化），但内容不同：
- 合规规则：判断是否违规（如"无视用户拒绝"）
- 归因规则：判断对话中什么因素导致没谈成（如"未介绍岗位亮点"、"语气过于机械"）

归因规则可以独立生成和更新，不需要改框架。

### 6.4 输入

- 非"有意向"对话的对话文本（直接从 Excel 原始数据获取，不依赖通道一结果）
- 意向结果标签（B/C/E/F/I 等）

归因分析是独立运行的，Worker 只需要对话文本和意向标签，不需要通道一的质检结果。

### 6.5 分析内容

1. **问题关联**：哪些问题类型和"无意向"强相关？
2. **轮次分析**：对话在哪个轮次开始偏离？
3. **高频模式**：哪些因素是"无意向"的高频原因？
4. **对比分析**：对比"有意向"组 vs"其他"组的对话模式差异

### 6.6 输出

独立 sheet 的归因报告，按意向结果类别分组统计：

**格式**：
| 意向结果 | 归因类别 | 占比 | 典型案例 | 改进建议 |
|----------|----------|------|----------|----------|

**示例**：
- B(不确定): 原因 a(未介绍岗位亮点) 占比 20%, 原因 b(未正面回答追问) 占比 15%...
- F(无意向): 原因 a(无视用户拒绝) 占比 25%, 原因 c(语气过于机械) 占比 18%...

每个意向结果类别下，列出导致该结果的原因类别及其占比，附带 3-5 个典型案例和改进建议。

## 8. Skill 结构

```
~/.agents/skills/auto-qc/
├── SKILL.md                        # Skill 主文件（流程定义 + Sub-agent 调度）
├── templates/
│   ├── worker-prompt.md            # Worker 打标模板（质检规则包注入）
│   ├── attribution-prompt.md       # 归因分析模板
│   └── attribution-rules.md        # 内置归因规则（5 Why 根因分析）
├── scripts/
│   ├── data_loader.py              # Excel 读取 + 对话预处理 + 列匹配 + 批次拆分 + 进度管理
│   └── report_writer.py            # 报告 Excel 输出 + 临时文件清理
└── requirements.txt                # Python 依赖（uv install 用）
```

### 8.1 用户交互

```
/auto-qc --data <excel路径> --rules <规则路径> [--no-attribution]
```

- `--data`: 源数据 Excel 路径（必需）
- `--rules`: 合规规则文件路径（必需，用户提供）
- `--no-attribution`: 关闭归因分析（默认开启）

**示例**：
- `/auto-qc --data data.xlsx --rules rules.md` → 合规检测 + 归因分析（默认全开）
- `/auto-qc --data data.xlsx --rules rules.md --no-attribution` → 仅合规检测
- `/auto-qc --data data.xlsx --attribution-only` → 仅归因分析（内置归因规则，不需要合规规则）

## 9. 错误处理

- Excel 读取失败 → 提示用户检查文件格式
- 规则解析失败 → 提示用户检查规则格式
- Worker Agent 超时/失败 → 自动重试（最多 3 次）
- 交叉验证差异率 > 5% → 标记为可疑，需要人工复核
- 进度文件损坏 → 从头开始或从最后一个完整批次恢复

## 10. 进度反馈与用户体验

### 10.1 进度文件

`progress.json` 实时记录当前状态：
```json
{
  "total_batches": 100,
  "completed_batches": 23,
  "current_batch": 24,
  "status": "running",  // running | cross-check | attribution | done | failed
  "completed_ids": [...],
  "failed_batches": [],
  "batch_status": {"1": "done", "2": "done", ...},
  "cross_check_results": {},
  "started_at": "2026-06-01T10:00:00",
  "updated_at": "2026-06-01T10:15:00"
}
```

### 10.2 断点续跑

Supervisor 每次启动时自动检测 `progress.json`：
- 存在且有未完成批次 → 提示用户："检测到上次中断的进度（已完成 X/Y 批），是否继续？"
- 用户确认后 → 从未完成的下一批继续，不重跑已完成批次
- 用户否认 → 从头开始，清空旧进度
- 用户不回复 → 默认继续（10 秒超时后自动恢复）

### 10.3 关键节点汇报

Supervisor 在以下节点输出简短汇报：
- "已加载 X 条对话，拆分为 Y 批"
- "已完成 10/100 批（10%）"
- "交叉验证中（抽样 N 条）"
- "归因分析中"
- "报告生成完成：xxx"

用户可以随时中断查看进度文件。

## 11. 失败重试策略

- Worker Agent 失败/超时/输出格式异常 → 自动重试，最多 3 次
- 3 次都失败的批次 → 记录到 `failed_batches.json`
- 不阻塞整体流程，最后汇总报告时提醒用户
- `failed_batches.json` 示例：
  ```json
  {
    "failed_batches": [
      {"batch_id": 3, "error": "timeout", "retries": 3},
      {"batch_id": 47, "error": "invalid_output", "retries": 3}
    ]
  }
  ```

## 15. 核心约束

**LLM 主导原则**：
- Worker 质检环节（通道一合规检测 + 通道二归因分析）必须由大模型主导完成
- 禁止用 Python 代码替代 LLM 做质检判断
- Python 仅负责 I/O 操作：Excel 读取、批次拆分、进度管理、报告输出
- 质检的"理解对话、判断违规、输出打标、归因分析"全部由 Claude 的 sub-agent 机制完成
- SKILL.md 的设计应引导 Claude 合理使用 sub-agent 分发任务，而非写代码自动处理

## 16. 对话预处理

### 16.1 原始数据格式

Excel 中对话文本为 JSON 格式，包含 TTS（AI）和 ASR（用户）轮次：
```json
[
  {"ttsResult": "你好，请问是张三吗？", "asrResult": "用户无应答"},
  {"ttsResult": "您好，我是猎聘这边的...", "asrResult": "是的哪位？"},
  ...
]
```

### 16.2 预处理流程

Python 负责将原始 JSON 转换为 LLM 易读的对话文本格式：
```
AI: 你好，请问是张三吗？
用户: 用户无应答

AI: 您好，我是猎聘这边的...
用户: 是的哪位？

AI: ...
用户: ...
```

预处理由 `data_loader.py` 中的 `preprocess_conversations()` 函数完成，输出为对话文本字符串，随批次数据一起分发给 Worker。

### 16.3 附加信息

预处理后的每条对话附带：
- `id`: 通话 ID
- `意向结果`: A/B/C/E/F/I 等
- `通话时长`: 从时间戳计算
- `轮次数`: TTS/ASR 对数

## 18. 容错机制

### 18.1 LLM JSON 输出容错

Worker 输出 JSON 时可能出现格式问题（多余逗号、缺引号、换行异常）。Python 层增加 `json_repair` 库做预处理：
1. 先用 `json_repair` 尝试修复不规范 JSON
2. 修复后尝试 `json.loads`
3. 仍失败则要求 Worker 重试（最多 3 次）

### 18.2 批次重试策略

- Worker Agent 失败/超时/输出格式异常 → 自动重试，最多 3 次
- 3 次都失败的批次 → 记录到 `failed_batches.json`
- 不阻塞整体流程，最后汇总报告时提醒用户

## 19. Skill 可移植性

### 19.1 打包结构

Skill 自包含，无需额外项目文件夹：
```
~/.agents/skills/auto-qc/
├── SKILL.md                    # Skill 主文件
├── templates/
│   ├── worker-prompt.md        # Worker 打标模板
│   ├── attribution-prompt.md   # 归因分析模板
│   └── attribution-rules.md    # 内置归因规则
├── scripts/
│   ├── data_loader.py          # Excel 读取 + 预处理 + 批次拆分
│   └── report_writer.py        # 报告输出 + 临时文件清理
└── requirements.txt            # Python 依赖（uv install 用）
```

### 19.2 分发方式

打包为 `.zip` 后，接收方只需：
1. 解压到 `~/.agents/skills/auto-qc/`
2. 打开 Claude Code，`/auto-qc` 即可使用
3. 首次使用时自动安装 Python 依赖（uv install）

### 19.3 SKILL.md 设计

SKILL.md 是 skill 的核心入口，包含：
- **触发方式**：用户输入 `/auto-qc --data <路径> --rules <路径>`
- **执行流程**：引导 Claude 作为 Supervisor，逐步完成规则解析→数据加载→批次拆分→Worker 分发→结果收集→报告输出
- **Sub-agent 调度**：明确指示 Claude 使用 `Agent` 工具启动多个 Worker，每次 5 个并发
- **Python 调用边界**：明确 Python 只负责 I/O（读 Excel、写 Excel、拆分数据），质检判断必须由 sub-agent 完成
- **错误处理**：遇到异常时的重试策略和用户提示
